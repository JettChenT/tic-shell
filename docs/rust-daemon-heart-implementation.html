tic-shell Rust Daemon and Heart Implementation

The Noctalia Codex pane now starts a Rust daemon that owns the shell-to-Codex bridge, starts Codex ACP, exposes a daemon MCP server, and runs the first Heart heartbeat system over niri/tiri window and workspace activity.

Entrypoint tic-daemon Config ~/.tic/config.toml Memory ~/.tic/memory/history/ MCP tic-daemon mcp

Runtime Ownership

The process tree has been inverted from "QML starts Bun, Bun starts CUA MCP processes" to "QML starts Rust, Rust starts Codex ACP, Codex sessions attach the daemon MCP".

Noctalia QML AgentBridge.qml starts a process and speaks the existing line-delimited JSON UI protocol.
tic-daemon Loads config, starts UI writer, starts ACP bridge, starts Heart supervisor.
codex-acp Managed under ~/.tic/codex-acp unless overridden.
Codex Session Per-workspace session with generated AGENTS.md and confined filesystem tools.
Daemon MCP tic-daemon mcp exposes emit_event plus CUA compatibility tools.

Implementation Map

Daemon Entrypoint

tic-daemon/src/main.rs dispatches between long-running daemon mode and MCP server mode.

Daemon mode loads config, starts ACP, starts Heart, and forwards QML stdin messages to the bridge.

ACP Bridge

tic-daemon/src/acp.rs ports the core Bun bridge behavior into Rust.

It handles initialization, workspace sessions, prompts, permission auto-allow, snapshots, and filesystem confinement.

Heart

tic-daemon/src/heart.rs owns the event bus and heartbeat loops.

It consumes niri events, cron ticks, screenshots, and emitted MCP updates.

MCP Server

tic-daemon/src/mcp.rs exposes emit_event and CUA-compatible tools.

Current CUA tool bodies use a compatibility command path, while the crate split allows direct library calls next.

CUA Split

cua/src/lib.rs now contains the previous implementation and cua/src/main.rs is a tiny CLI wrapper.

This makes the CUA code linkable from Rust consumers.

Noctalia Wiring

AgentBridge.qml now runs cargo run --quiet --bin tic-daemon from TIC_SHELL_ROOT.

The QML message shape remains unchanged for the first migration.

Daemon Startup Flow

1
QML starts the daemon. Noctalia launches cargo run --quiet --bin tic-daemon with TIC_SHELL_ROOT.
2
Config is loaded or created. Defaults are written to ~/.tic/config.toml if the file is absent.
3
Codex ACP starts. The daemon uses TIC_CODEX_ACP_COMMAND or the managed @zed-industries/codex-acp@0.13.0 install under ~/.tic/codex-acp.
4
Workspace session is created. The daemon sends initialize, then session/new with a generated workspace root and the daemon MCP server.
5
Heart starts in parallel. The supervisor subscribes to niri event-stream, screenshot ticks, workspace interval ticks, and the MCP event log.

Event Bus

Heart normalizes four categories of events. The abstraction is intentionally small so additional heartbeat loops can subscribe to the same bus later.

VariantProducerConsumer Logic
Event::TiriEvent niri msg --json event-stream Creates, updates, and closes window/workspace heartbeat state.
Event::Cron Tokio interval tasks Drives screenshot ticks and L2 workspace interval triggers.

L1 Window Heartbeat

State

Each tracked niri window has a WindowHeartbeat with the latest window metadata, accepted screenshot buffer, trigger counters, timestamps, and previous perceptual hash.

WindowHeartbeat {
  window,
  buffer,
  accepted_since_trigger,
  initialized,
  started,
  last_trigger,
  last_hash,
}

Trigger Logic

Initial trigger fires after 10 accepted screenshots or 20s. Ongoing trigger fires after 30 accepted screenshots or 60s. Close trigger fires on WindowClosed.

All values are defaults from config, not hardcoded policy.

StepBehaviorResult
Observe window WindowOpenedOrChanged or initial bootstrap calls ensure_window. L1 state exists for the window id.
Capture screenshot Screenshot cron writes a PNG through compositor-native cua-screenshot-window. Event::Screenshot enters the Heart bus.
Filter img_hash compares the new image to last_hash. Insignificant screenshots are deleted and not buffered.
Buffer Accepted screenshots are capped by buffer_max, default 100. Overflow images are dropped and removed from disk.
Prompt Codex Daemon sends a Heart prompt to a heart:window:<id> ACP session. Codex updates a 10min-window-summary.md file and calls emit_event.

L2 Workspace Heartbeat

L2 aggregates L1 window updates for each workspace. The first update from any window triggers immediately, the interval trigger defaults to 60min, and close triggers finalize the workspace summary.

L1 Codex Writes window summary and calls emit_event.
events.jsonl Daemon MCP appends structured event lines.
Event Tailer Heart reads new log lines and emits HeartUpdate.
L2 State Workspace buffers pending updates and tracks first seen windows.
L2 Codex Updates 60min-workspace-summary.md and emits workspace_heartbeat_l2_update.

Configuration Surface

The daemon creates this TOML file on first run. Constants from the Heart spec are expressed as defaults so runtime behavior can be tuned without code changes. 

[daemon]
data_root = "/home/jettc/.tic"
memory_root = "/home/jettc/.tic/memory"
workdir_root = "/run/user/1000/tic-shell/codex-workspaces"

[heartbeat.window-l1]
enabled = true
screenshot_interval = "2s"
initial_screenshot_count = 10
initial_timeout = "20s"
ongoing_screenshot_count = 30
buffer_max = 100
summary_duration_label = "10min"

[heartbeat.window-l1.ongoing]
interval = "60s"

[heartbeat.workspace-l2]
enabled = true
interval = "60min"
summary_duration_label = "60min"

[heartbeat.screenshot-diff]
enabled = true
algorithm = "gradient-hash"
threshold = 8

Live Runtime

Noctalia is running from shell/noctalia.

tic-daemon is a child of Quickshell, and codex-acp is a child of tic-daemon.

Layer surfaces still include tic-workspace-eDP-1 and tic-workspace-exclusion-left-eDP-1.

Known implementation boundary: the daemon MCP currently exposes CUA tools through a compatibility command path. The cua code has been moved into a library-capable shape, so the next tightening step is to call those routines directly from tic-daemon instead of shelling through the CLI.
